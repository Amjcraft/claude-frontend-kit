# Project Conventions

Standards for file structure, imports, and code organization in this project.

## Import Ordering

Strict import order — enforce this in every file:

```tsx
// 1. React and Next.js
import { useState, useCallback } from "react"
import { useRouter } from "next/navigation"
import Image from "next/image"

// 2. Third-party libraries
import { z } from "zod"
import { useForm } from "react-hook-form"

// 3. Path aliases (internal modules)
import { Button } from "@/components/ui/button"
import { useAuth } from "@/hooks/use-auth"
import { formatDate } from "@/lib/utils"

// 4. Relative imports
import { SubmitButton } from "./submit-button"

// 5. Type-only imports (last)
import type { User } from "@/types/user"
```

## Path Aliases

Always use `@/` aliases. Never use `../../../` relative paths for anything beyond the immediate directory.

```tsx
// ✅
import { Button } from "@/components/ui/button"
import { db } from "@/lib/db"

// ❌
import { Button } from "../../../components/ui/button"
```

Standard aliases configured in `tsconfig.json`:
- `@/` → `src/`
- `@/components` → `src/components/`
- `@/lib` → `src/lib/`
- `@/hooks` → `src/hooks/`
- `@/types` → `src/types/`

## File Creation Checklist

### New Component
- [ ] File at `src/components/{category}/{name}/{name}.tsx`
- [ ] Test at `src/components/{category}/{name}/{name}.test.tsx`
- [ ] Barrel export at `src/components/{category}/{name}/index.ts`
- [ ] Types exported from the component file or a co-located `types.ts`
- [ ] Uses `cn()` for class merging
- [ ] Has `displayName` if using `forwardRef`

### New Page (App Router)
- [ ] File at `src/app/{route}/page.tsx`
- [ ] Layout if needed: `src/app/{route}/layout.tsx`
- [ ] Loading state: `src/app/{route}/loading.tsx`
- [ ] Error boundary: `src/app/{route}/error.tsx`
- [ ] Metadata exported (`export const metadata`)

### New Hook
- [ ] File at `src/hooks/use-{name}.ts`
- [ ] Named export (not default)
- [ ] Types defined inline or in co-located types file
- [ ] Unit test at `src/hooks/use-{name}.test.ts`

## Environment Variables

- Every env var used in code must exist in `.env.example` with a placeholder
- Client-accessible vars use `NEXT_PUBLIC_` prefix
- Access server-only vars only in Server Components, API routes, or server actions
- Never access `process.env` directly in client components — use a typed config module

```ts
// src/lib/config.ts — typed env access
export const config = {
  apiUrl: process.env.NEXT_PUBLIC_API_URL!,
  // server-only
  databaseUrl: process.env.DATABASE_URL!,
} as const
```

## Error Handling

### Server Actions / API Routes
```ts
// Return typed result objects — don't throw across boundaries
type ActionResult<T> =
  | { success: true; data: T }
  | { success: false; error: string }
```

### Client Components
- Use React Error Boundaries for unexpected rendering errors
- Handle async errors with try/catch in event handlers
- Show user-facing error messages via toast or inline error states — never raw error strings

## Naming Conventions

| Thing | Convention | Example |
|-------|-----------|---------|
| Components | PascalCase | `UserAvatar` |
| Hooks | camelCase with `use` prefix | `useUserSession` |
| Utils/helpers | camelCase | `formatCurrency` |
| Types/interfaces | PascalCase | `UserProfile` |
| Constants | SCREAMING_SNAKE_CASE | `MAX_RETRY_COUNT` |
| Files | kebab-case | `user-avatar.tsx` |
| CSS variables | kebab-case | `--color-primary` |

## shadcn/ui Rules

- Never modify files in `src/components/ui/` directly — these are generated by shadcn CLI
- Extend via wrapper components or Tailwind class overrides via `className` prop
- Run `pnpm dlx shadcn@latest add {component}` to add new primitives
